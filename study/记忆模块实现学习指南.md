# 记忆模块实现学习指南

> 目标：理解 assistant 项目的记忆模块如何存储长期信息、如何暴露给 Agent 调用、如何自动从长对话中提炼记忆，以及这些记忆如何进入后续对话上下文。

## 一、先建立全局图景

记忆模块不是单个工具，而是一套由「数据模型 + 持久化管理 + Agent 工具 + 自动整理 + Prompt 注入」组成的闭环。

```text
MainWindow 启动
    |
    +-- 创建 MemoryManager
    |      |
    |      +-- 确保 SQLite memories 表存在
    |
    +-- 创建 ToolContext(extra={"memory_mgr": ...})
    |      |
    |      +-- load_builtin_tools 自动发现 app/tools/memory.py
    |      +-- 注册 save_memory / recall_memory / forget_memory
    |
    +-- 创建 Consolidator
    |      |
    |      +-- 对话过长时压缩旧消息
    |      +-- 把摘要保存为 archive 记忆
    |
    +-- 创建 Dream
           |
           +-- 每小时后台运行
           +-- 从 archive 摘要中提炼结构化长期记忆
```

每次请求模型前：

```text
MainWindow._build_api_messages()
    |
    +-- 拼接用户 system prompt
    +-- 拼接当前时间和内置工具说明
    +-- MemoryManager.build_memory_context(session_id)
           |
           +-- 查询 global 记忆
           +-- 查询当前 session 记忆
           +-- 排除 archive 原始摘要
           +-- 生成 "## 长期记忆" 文本块
```

一句话概括：

```text
工具调用负责显式记住，Consolidator 负责保存对话摘要，Dream 负责把摘要整理成真正的长期记忆，MainWindow 负责把长期记忆注入下一轮 system prompt。
```

## 二、核心文件地图

| 文件 | 角色 | 重点 |
|------|------|------|
| `app/models/memory.py` | 数据模型 | 记忆分类、范围、字段结构 |
| `app/core/memory_manager.py` | 持久化层 | 建表、CRUD、查询、上下文构建 |
| `app/tools/memory.py` | Agent 工具 | save/recall/forget 三个内置工具 |
| `app/core/consolidator.py` | 对话压缩 | 超长上下文摘要，保存为 archive |
| `app/core/dream.py` | 自动整理 | 从 archive 提取长期记忆 |
| `app/ui/main_window.py` | 组装入口 | 初始化、定时器、上下文注入 |
| `app/core/agent_loop.py` | 工具执行 | 注入 `_session_id` 给记忆工具 |
| `app/core/constants.py` | Prompt 规则 | 告诉模型何时使用记忆工具 |

## 三、数据模型：Memory 是什么

入口文件：

```text
app/models/memory.py
```

### 1. 分类 category

`MemoryCategory` 定义了五类记忆：

| category | 含义 | 是否直接注入 Prompt |
|----------|------|--------------------|
| `personality` | AI 人设、行为风格 | 是 |
| `user` | 用户身份、偏好、习惯 | 是 |
| `project` | 项目决策、技术选型、架构约定 | 是 |
| `fact` | 具体事实，如路径、配置、流程 | 是 |
| `archive` | Consolidator 生成的对话摘要 | 否，作为 Dream 原料 |

关键点：`archive` 不是最终长期记忆，它是中间产物。真正给模型看的长期记忆会排除 `archive`。

### 2. 范围 scope

`MemoryScope` 只有两个值：

| scope | 含义 |
|-------|------|
| `global` | 所有 session 共享 |
| `session` | 只对指定 session 可见 |

`session` 记忆必须配合 `session_id` 使用。工具参数里虽然允许模型选择 scope，但当前 session ID 不是模型填写的，而是 `AgentLoop` 在执行工具前注入的。

### 3. 主要字段

`Memory` dataclass 的核心字段：

| 字段 | 用途 |
|------|------|
| `content` | 记忆正文，一般是一句简洁事实 |
| `category` | 分类，用于筛选和展示 |
| `scope` | 全局或会话级 |
| `session_id` | 会话级记忆归属 |
| `source` | 来源：`tool` / `consolidator` / `dream` |
| `importance` | 重要性 1-10，影响注入排序 |
| `is_processed` | archive 是否已被 Dream 处理 |
| `created_at` / `updated_at` | 时间戳 |
| `expired_at` | 过期时间，目前只有清理方法使用 |

学习重点：

```text
category 决定这条信息是什么
scope 决定谁能看到
source 决定它从哪里来
importance 决定注入优先级
is_processed 只服务 archive -> Dream 的流程
```

## 四、持久化层：MemoryManager

入口文件：

```text
app/core/memory_manager.py
```

`MemoryManager` 是记忆模块的核心服务层。它接收一个 `DatabaseManager`，并复用项目已有的 SQLite 数据库连接。

### 1. 初始化和建表

`MemoryManager.__init__()` 会调用 `_ensure_table()`，确保 `memories` 表存在：

```sql
CREATE TABLE IF NOT EXISTS memories (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    content TEXT NOT NULL,
    category TEXT NOT NULL DEFAULT 'fact',
    scope TEXT NOT NULL DEFAULT 'global',
    session_id TEXT,
    source TEXT NOT NULL DEFAULT 'tool',
    importance INTEGER NOT NULL DEFAULT 5,
    is_processed INTEGER NOT NULL DEFAULT 0,
    created_at REAL NOT NULL,
    updated_at REAL NOT NULL,
    expired_at REAL
);
```

同时创建三个索引：

| 索引 | 用途 |
|------|------|
| `idx_memories_scope` | 按 scope/session_id 查询上下文记忆 |
| `idx_memories_category` | 按分类筛选 |
| `idx_memories_is_processed` | Dream 查找未处理 archive |

注意：`memories` 表不是在 `DatabaseManager._create_tables()` 里创建的，而是在 `MemoryManager` 自己初始化时创建。这让记忆模块保持独立，但也意味着必须创建 `MemoryManager` 后表才会出现。

### 2. CRUD 方法

常用方法：

| 方法 | 作用 |
|------|------|
| `add()` | 新增记忆 |
| `update()` | 更新正文、分类、重要性、处理状态 |
| `delete()` | 删除指定 ID |
| `get_by_id()` | 按 ID 查询 |
| `count()` | 统计数量 |

`add()` 会自动填充 `created_at` 和 `updated_at`，并返回一个 `Memory` 对象。

### 3. 查询方法

| 方法 | 查询范围 |
|------|----------|
| `get_global(category=None)` | 所有全局记忆，可按分类过滤 |
| `get_for_session(session_id)` | 指定 session 的局部记忆 |
| `get_context_memories(session_id)` | 准备注入 Prompt 的记忆 |
| `get_unprocessed_archives()` | Dream 待处理的 archive |

最重要的是 `get_context_memories()`：

```text
category != 'archive'
AND (
  scope = 'global'
  OR (scope = 'session' AND session_id = 当前会话)
)
ORDER BY importance DESC, updated_at DESC
```

这说明：

- archive 不直接进入 system prompt。
- 全局记忆总是可见。
- session 记忆只在当前会话可见。
- 重要性高、更新更晚的记忆优先。

### 4. 构建 Prompt 文本

`build_memory_context(session_id, max_chars=4000)` 会生成：

```markdown
## 长期记忆

- [project] 项目使用 PyQt6 无边框透明浮窗
- [user] 用户偏好中文回复
```

它按顺序累加记忆，超过 `max_chars` 就停止。这是一个简单但有效的上下文预算控制。

## 五、手动记忆链路：save / recall / forget

入口文件：

```text
app/tools/memory.py
```

这里定义了三个内置工具，都会通过 `ToolContext.extra["memory_mgr"]` 拿到同一个 `MemoryManager`。

### 1. save_memory

用途：当用户明确要求“记住某事”，或者系统规则要求保存项目决策时，模型调用它。

参数：

| 参数 | 来源 | 说明 |
|------|------|------|
| `content` | 模型填写 | 简洁明确的记忆正文 |
| `category` | 模型填写 | personality/user/project/fact |
| `scope` | 模型填写 | global/session |
| `importance` | 模型填写 | 1-10 |
| `_session_id` | AgentLoop 注入 | 不暴露给模型 |

执行逻辑：

```text
SaveMemoryTool.execute()
    |
    +-- 读取 content/category/scope/importance
    +-- 读取 AgentLoop 注入的 _session_id
    +-- MemoryManager.add(...)
    +-- 返回新记忆 ID
```

如果 scope 是 `session`，才会写入 `session_id`；如果 scope 是 `global`，`session_id` 会被置为 `None`。

### 2. recall_memory

用途：用户问“你记住了什么？”时查看记忆。

特点：

- `read_only=True`，所以可以被 AgentLoop 放进并发只读工具批次。
- 支持按 `category` 和 `scope` 过滤。

当前实现有一个细节：如果不传 scope，默认只查 global；不会同时返回 global + 当前 session。真正“上下文注入”使用的是 `get_context_memories()`，和 `recall_memory` 的查询策略不完全一样。

### 3. forget_memory

用途：按 ID 删除记忆。

执行逻辑很直接：

```text
ForgetMemoryTool.execute()
    |
    +-- 读取 memory_id
    +-- MemoryManager.delete(memory_id)
    +-- 根据 rowcount 返回成功或不存在
```

### 4. 工具如何被注册

`app/tools/loader.py` 会扫描 `app/tools/*.py`，找到所有 `BuiltinTool` 子类，然后调用 `create(ctx)` 创建实例。

记忆工具的依赖来自这里：

```text
MainWindow._init_tools()
    |
    +-- self._memory_mgr = MemoryManager(self._notes.db)
    +-- ToolContext(extra={"memory_mgr": self._memory_mgr})
    +-- load_builtin_tools(ctx, self._registry)
```

这是一种依赖注入模式：工具自己不创建数据库，不关心 UI，只拿到执行所需的服务对象。

### 5. _session_id 如何进入工具

`AgentLoop` 执行工具前，会把当前 session ID 塞进参数：

```text
if self._session_id:
    ai_args["_session_id"] = self._session_id
```

这个字段不在工具 schema 里，所以模型看不到，也不能伪造。它解决了 session 级记忆的归属问题。

## 六、自动记忆链路一：Consolidator 对话压缩

入口文件：

```text
app/core/consolidator.py
```

Consolidator 解决的是长对话上下文过长的问题。它不是直接提炼长期记忆，而是先把旧消息压缩成 archive。

### 1. 什么时候触发

`MainWindow` 在每次启动 AgentLoop 前调用：

```text
MainWindow._send_message()
    |
    +-- session = self._sessions.get(sid)
    +-- original_msgs = session.messages[:-1]
    +-- compressed = self._consolidator.maybe_consolidate(original_msgs, sid)
    +-- 如果压缩后消息变少，保存 session
```

这里排除了刚创建的空 assistant 占位消息。

### 2. token 估算

`_estimate_tokens()` 是粗略估算：

```text
中文：约 1.5 字 / token
英文：约 4 字符 / token
```

默认配置：

```text
max_context_tokens = 60000
threshold = max_context_tokens * 0.7 = 42000
consolidation_ratio = 0.5
target_tokens = 30000
```

当估算 token 超过阈值时，Consolidator 会从最旧消息开始压缩，至少保留最近 4 条消息。

### 3. 压缩过程

```text
maybe_consolidate(messages, session_id)
    |
    +-- 估算总 token
    +-- 未超阈值：原样返回
    +-- 超阈值：切分 to_compress / to_keep
    +-- _summarize(to_compress)
    +-- MemoryManager.add(category=archive, scope=session, source=consolidator)
    +-- 返回 [摘要 system 消息] + to_keep
```

摘要会同时进入两个地方：

1. 当前 session 消息列表：作为 system 消息替代旧消息，保持对话连续。
2. `memories` 表：作为 `archive` 记忆，等待 Dream 后续处理。

### 4. 失败兜底

如果 LLM 摘要失败，Consolidator 不会直接中断对话，而是用 fallback：

```text
取每条旧消息前 100 字符，最多保留最后 20 条
```

这保证“压缩失败”不会变成“对话无法继续”。

## 七、自动记忆链路二：Dream 长期记忆提炼

入口文件：

```text
app/core/dream.py
```

Dream 是后台记忆整理器。它读取未处理的 archive 摘要，让 LLM 判断哪些内容值得变成长期记忆。

### 1. 初始化和定时运行

`MainWindow._init_memory()` 创建 Dream，并设置定时器：

```text
self._dream_timer = QTimer(self)
self._dream_timer.timeout.connect(self._run_dream)
self._dream_timer.start(3600_000)
```

也就是每小时触发一次。

`_run_dream()` 会开后台线程：

```text
threading.Thread(target=self._dream.run, daemon=True).start()
```

这样不会阻塞 PyQt UI 主线程。

### 2. Dream 的完整流程

```text
Dream.run()
    |
    +-- MemoryManager.get_unprocessed_archives()
    |      |
    |      +-- category='archive'
    |      +-- is_processed=0
    |
    +-- MemoryManager.get_global()
    |
    +-- 构建 _DREAM_PROMPT
    |
    +-- 调用 LLM
    |
    +-- 解析 JSON 指令
    |
    +-- 执行 ADD / UPDATE / DELETE
    |
    +-- mark_archives_processed(ids)
```

### 3. Dream Prompt 要求 LLM 输出指令

Dream 不让模型直接写数据库，而是要求输出 JSON 数组：

```json
[
  {"action": "ADD", "content": "记忆内容", "category": "project", "importance": 8},
  {"action": "UPDATE", "id": 123, "content": "更新后的内容", "importance": 7},
  {"action": "DELETE", "id": 456, "reason": "已过时"}
]
```

然后 Python 代码执行这些指令。

这种设计的好处：

- LLM 只负责判断和生成结构化计划。
- 数据库修改仍由确定性代码完成。
- 可以对 category 和 importance 做校验。

### 4. 指令执行细节

`ADD`：

- category 只允许 `personality/user/project/fact`。
- scope 固定为 `global`。
- source 固定为 `dream`。
- importance 限制在 1-10。

`UPDATE`：

- 按 ID 更新 content 和 importance。
- 当前不会更新 category。

`DELETE`：

- 按 ID 删除。

### 5. archive 处理标记

Dream 成功执行后，会调用：

```text
MemoryManager.mark_archives_processed(archive_ids)
```

这会把 archive 的 `is_processed` 设为 1，避免下次重复处理。

如果 Dream 调用或解析出错，`run()` 返回 False，不标记 processed。这样下次还有机会重试。

## 八、上下文注入链路：记忆如何影响下一轮回答

入口文件：

```text
app/ui/main_window.py
```

每轮请求模型前，`MainWindow._build_api_messages()` 会构建 API messages。

System Prompt 拼接顺序：

```text
1. 会话级 system_prompt
2. 全局配置 system_prompt
3. 默认 DEFAULT_USER_PROMPT
4. 当前时间信息 get_current_datetime_info()
5. BUILTIN_TOOLS_INSTRUCTION
6. 长期记忆 build_memory_context()
```

也就是说，记忆最后追加到 system prompt 中：

```text
full_system_prompt += "\n\n" + memory_context
```

这个顺序的含义：

- 用户配置和系统工具规则先定义基本行为。
- 长期记忆作为额外上下文补充。
- 记忆不是单独 message，而是合并进 system prompt。

## 九、内置 Prompt 如何引导模型使用记忆工具

入口文件：

```text
app/core/constants.py
```

`BUILTIN_TOOLS_INSTRUCTION` 里有记忆规则：

```text
【记忆】
- 当用户要求你记住某些信息时，使用 save_memory 工具保存。
- 当用户询问你记住了什么时，使用 recall_memory 工具查看。
- 当用户要求你忘记某条信息时，使用 forget_memory 工具删除。
- 分类：user（用户偏好）、project（项目决策）、fact（具体事实）、personality（AI行为风格）。
- 范围：global（所有会话可见）、session（仅当前会话可见）。
```

这部分不是给用户看的 UI 文案，而是注入给模型的行为约束。它让模型知道什么时候应该主动调用记忆工具。

## 十、三条典型数据流

### 1. 用户明确要求“记住”

```text
用户：记住这个项目拖动窗口要用 startSystemMove()
    |
    +-- 模型看到工具说明
    +-- 调用 save_memory
    +-- AgentLoop 注入 _session_id
    +-- SaveMemoryTool.execute()
    +-- MemoryManager.add(source=tool)
    +-- 写入 memories 表
    |
下一轮请求：
    |
    +-- build_memory_context()
    +-- 记忆注入 system prompt
```

### 2. 用户询问“你记住了什么”

```text
用户：你记住了哪些项目规则？
    |
    +-- 模型调用 recall_memory(category=project)
    +-- RecallMemoryTool.execute()
    +-- MemoryManager.get_global(category=project)
    +-- 返回记忆列表
    +-- 模型整理后回复用户
```

注意：如果要查 session 级记忆，需要传 `scope=session`，否则默认只查 global。

### 3. 长对话自动沉淀长期记忆

```text
对话越来越长
    |
    +-- MainWindow 调用 Consolidator
    +-- token 超过阈值
    +-- 旧消息被摘要
    +-- 摘要写入 memories(category=archive, source=consolidator)
    |
每小时 Dream 运行
    |
    +-- 读取未处理 archive
    +-- 连同已有 global 记忆发给 LLM
    +-- LLM 输出 ADD / UPDATE / DELETE
    +-- 写入真正长期记忆 source=dream
    +-- archive 标记 processed
```

## 十一、设计取舍和优点

### 1. 显式记忆和自动记忆分离

`save_memory` 是明确、即时、可控的。

`Consolidator + Dream` 是后台、延迟、自动的。

这种分离避免了每轮对话都让 LLM 做沉重的记忆整理，也避免了把所有对话原文都直接当长期记忆。

### 2. archive 作为中间层

archive 的作用类似“原料缓存”：

```text
长对话原文 -> 摘要 archive -> Dream 提炼 -> 长期记忆
```

优点：

- 降低 Dream 的输入长度。
- 可以失败重试。
- 可以追踪哪些摘要已经处理。

### 3. source 字段让来源可追踪

`source` 能区分：

- 用户明确保存的记忆：`tool`
- 对话压缩摘要：`consolidator`
- 后台整理出来的长期记忆：`dream`

调试记忆质量时，这个字段很关键。

### 4. scope 支持全局和会话隔离

全局记忆适合项目规则、用户长期偏好。

会话记忆适合只在当前任务里有效的信息。

### 5. importance 控制上下文预算

`build_memory_context()` 默认最多注入 4000 字符。排序按 `importance DESC, updated_at DESC`，所以重要记忆优先进入 prompt。

## 十二、当前实现的注意点

### 1. recall_memory 默认不返回 session 记忆

如果 `scope` 不传，`RecallMemoryTool` 当前走 `get_global()`。这和上下文注入的“global + 当前 session”不同。

如果希望“查看当前可见全部记忆”，可以考虑扩展一个 `scope=all`，或让不传 scope 时使用 `get_context_memories(session_id)`。

### 2. Dream 只读取 global 作为已有记忆

Dream 构建 prompt 时调用的是 `get_global()`。这意味着它主要维护全局长期记忆，不会把 session 级记忆作为已有记忆上下文。

这是合理的默认策略，但如果未来要做“会话级 Dream”，需要单独设计。

### 3. Dream 新增记忆固定为 global

`ADD` 指令执行时 scope 固定为 `MemoryScope.GLOBAL`。所以自动沉淀出来的长期记忆都会跨 session 可见。

如果有些 archive 只应该留在当前 session，需要在 Dream 指令协议里加入 scope，并在执行层做限制。

### 4. expire_stale 目前没有自动调度

`MemoryManager.expire_stale()` 已经实现，但当前没有看到定时调用。`expired_at` 字段也没有在 `add()` 参数中暴露。

也就是说“过期记忆”机制是预留能力，不是完整闭环。

### 5. memories 表由 MemoryManager 创建

如果写测试或脚本直接查数据库，要先实例化 `MemoryManager`，否则旧数据库里可能还没有 `memories` 表。

## 十五、如果要扩展这个模块

### 1. 增加记忆管理 UI

可以基于 `MemoryManager.get_global()` 和 `get_for_session()` 做一个面板：

- 列表展示 content/category/scope/source/importance。
- 支持编辑 content 和 importance。
- 支持删除。
- 支持按 source/category/scope 过滤。

### 2. 增加语义检索

当前记忆注入是按重要性和更新时间排序，不做语义相关性匹配。

可扩展方向：

```text
memories 表增加 embedding
用户当前问题 -> embedding
按相似度召回相关记忆
再按 importance 混合排序
```

这样可以减少无关记忆占用 prompt。

### 3. 改进 recall_memory

建议增加一种查询模式：

```text
scope = "visible"
```

含义：返回当前 session 实际可见的记忆，也就是 `get_context_memories(session_id)`。

### 4. 增加 Dream 审计日志

Dream 当前只执行指令，没有保存“为什么这么做”。

可扩展：

- 保存原始 LLM response。
- 保存每条 ADD/UPDATE/DELETE 的 reason。
- 在 UI 中展示 Dream 处理记录。

### 5. 完善过期机制

当前 `expired_at` 和 `expire_stale()` 是半成品能力。

可补齐：

- `save_memory` 支持 ttl 或 expired_at。
- Dream 可给低重要性事实设置过期时间。
- MainWindow 定时调用 `expire_stale()`。

## 十六、理解这套设计的关键句

```text
MemoryManager 是唯一直接操作 memories 表的服务层。
memory.py 里的工具只是 Agent 访问 MemoryManager 的入口。
Consolidator 不生产最终长期记忆，只生产 archive 摘要。
Dream 把 archive 转换成长期可注入的 global 记忆。
MainWindow 在每次请求模型前，把长期记忆拼进 system prompt。
```

掌握这五句话，就能从任意入口追踪记忆模块的完整行为。

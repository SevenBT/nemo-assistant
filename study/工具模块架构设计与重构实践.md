# 工具模块架构设计与重构实践

> 基于 assistant 与 nanobot 两个项目的工具系统对比分析，完成 Phase 1 + Phase 2 重构。

## 一、核心知识点

### 1. 两种工具架构范式

| 维度 | assistant（插件式） | nanobot（内聚式） |
|------|-------------------|-----------------|
| 工具定义 | manifest.json + tool.py 脚本 | Python 类继承 ABC |
| 执行模型 | 子进程隔离（subprocess） | 异步 in-process（async/await） |
| 注册机制 | 目录扫描 + JSON 解析 | pkgutil 模块扫描 + entry_points 插件 |
| 依赖管理 | 运行时 pip install 到隔离 site-packages | 项目 pyproject.toml 统一声明 |
| 用户扩展 | 支持用户通过 UI 创建/管理工具 | 不支持，开发者预定义 |
| 任意执行 | run_python 工具（Python 代码片段） | exec 工具（任意 shell 命令） |

**设计取舍**：assistant 面向非技术用户，工具有明确 schema 让 LLM 精准调用；nanobot 面向开发者，exec 覆盖一切。

### 2. 统一工具协议（Protocol）

用 Python Protocol（鸭子类型）定义统一接口，不强制继承：

```python
@runtime_checkable
class Tool(Protocol):
    @property
    def name(self) -> str: ...
    @property
    def read_only(self) -> bool: ...
    @property
    def enabled(self) -> bool: ...
    def to_openai_function(self) -> dict: ...
    def execute(self, params: dict) -> dict: ...
```

选择 Protocol 而非 ABC 的原因：外部脚本工具是包装类，不需要强制继承关系。

### 3. 并发执行策略

**批次分区算法**：连续的 read_only 工具合并为并发批次，有副作用工具单独串行。

```
[web_search, fetch_url, save_file, read_file]
→ [concurrent(web_search, fetch_url), serial(save_file), concurrent(read_file)]
```

使用 `ThreadPoolExecutor`（因为工具执行是 subprocess 阻塞调用，不是 async）。

### 4. 依赖注入模式

内置工具需要 NoteManager、SchedulerManager 等应用对象，通过构造函数注入：

```python
class CreateNoteTool:
    def __init__(self, note_mgr: NoteManager, on_created: Callable):
        self._notes = note_mgr  # 持有引用，后续 execute 时直接使用
```

main_window 负责组装（它拥有这些实例），创建后注册到 ToolManager，之后 AgentLoop 调用时自动使用。

### 5. 参数来源系统（assistant 独有）

三种来源，解析优先级：manual > ai > config > default

| source | 含义 | 示例 |
|--------|------|------|
| ai | LLM 填充，暴露在 function schema 中 | query、code |
| config | 应用配置注入，对 LLM 隐藏 | api_key、save_dir |
| manual | 运行时弹 UI 让用户输入 | 敏感操作确认 |

### 6. 结果处理增强

- **截断**：超过 8000 字符截断，追加提示告知 LLM
- **错误增强**：status=error 时追加引导语，促使 LLM 换方法重试
- **参数校验**：执行前校验 required + type + enum，失败直接返回错误不执行

### 7. 沙箱机制（nanobot）

两层防护：
- **应用层**（跨平台）：命令黑名单（rm -rf、format 等）、路径穿越拦截、内网 URL 拦截
- **内核层**（仅 Linux）：bubblewrap 做 namespace 隔离，工作区读写、系统目录只读、配置目录遮盖

assistant 当前无沙箱，但子进程隔离提供了基本的崩溃隔离。

### 8. nanobot 工具依赖不需要动态安装的原因

nanobot 工具是项目自身代码的一部分，所有依赖在 pyproject.toml 中声明，`pip install nanobot-ai` 时一次性装好。不存在"用户创建新工具带来未知依赖"的场景。

assistant 的 ToolDependencyManager 存在是因为工具集开放，用户可以随时创建带任意依赖的新工具。

## 二、重构实践记录

### 完成的改造

**Phase 1（快速增强）**：
- 工具并发执行（ThreadPoolExecutor, max_workers=4）
- 结果大小限制和截断（8000 字符）
- 错误结果追加引导提示
- 外部工具参数校验（required + type + enum）

**Phase 2（统一接口）**：
- 定义 Tool Protocol（`app/core/tool_protocol.py`）
- BuiltinToolHandler 拆分为 8 个独立 Tool 类
- ToolManager 重写为统一 registry（ScriptTool 包装外部脚本）
- AgentLoop 移除双轨分支，构造函数简化

### 关键设计决策

1. **彻底重构不留兼容包袱**：直接删除 BuiltinToolHandler 和 BUILTIN_TOOLS 列表，不写适配层
2. **ScriptTool 包装类**：将 ToolDefinition + subprocess 执行逻辑封装为满足 Tool Protocol 的类
3. **ToolManager.get_tools() 保留**：返回 ToolDefinition 列表供 UI toolbox_panel 使用（UI 层需要访问 version、author 等元数据）

## 三、会话中的问题记录

1. **Q: 依赖通过构造函数注入是什么意思？**
   - A: 工具运行需要的外部对象（NoteManager、SchedulerManager）不由工具自己创建，而是由拥有这些对象的 main_window 在构造时传入，工具持有引用后续直接使用。

2. **Q: nanobot 有沙箱机制吗？**
   - A: 有两层——应用级命令守卫（跨平台）+ bubblewrap 内核隔离（仅 Linux）。

3. **Q: nanobot 使用工具为什么不需要安装依赖？**
   - A: 工具是项目自身代码，依赖在 pyproject.toml 中统一声明，安装项目时一起装好。

4. **Q: nanobot 不支持任意脚本执行吗？**
   - A: 支持，通过 exec 工具执行任意 shell 命令。但不支持"用户创建命名工具"的概念，设计面向开发者而非终端用户。

# 工具模块学习路线

> 目标：从主体框架到具体实现，理解 assistant 项目里的工具系统如何被加载、注册、暴露给模型、执行并回传结果。

## 一、先建立全局图景

工具模块的核心不是某一个具体工具，而是一条完整链路：

```text
MainWindow 启动
    |
    +-- 创建 ToolContext
    |      |
    |      +-- config
    |      +-- note_mgr
    |      +-- scheduler
    |      +-- ai_client
    |      +-- UI 回调
    |
    +-- 创建 ToolRegistry
           |
           +-- 加载内置工具
           |      |
           |      +-- 扫描 app/tools/*.py
           |      +-- 找到 BuiltinTool 子类
           |      +-- 调用 Tool.create(ctx)
           |      +-- registry.register(tool)
           |
           +-- 加载用户脚本工具
                  |
                  +-- 扫描 data/user_tools/
                  +-- 读取 manifest.json
                  +-- 创建 ScriptToolAdapter
                  +-- registry.register(adapter)
```

运行时调用链路：

```text
AgentLoop
    |
    +-- registry.get_openai_functions()
    |      |
    |      +-- 把工具列表交给 AIClient.chat_stream()
    |
    +-- 模型返回 tool_call
           |
           +-- 判断工具 read_only
           |      |
           |      +-- read_only=True  -> 可并发执行
           |      +-- read_only=False -> 串行执行
           |
           +-- registry.execute(name, params)
                  |
                  +-- 查找工具
                  +-- cast_params()
                  +-- validate_params()
                  +-- tool.execute(params)
                  +-- format_result()
                  +-- 作为 tool 消息回传给模型
```

一句话概括：

```text
MainWindow 负责组装系统
ToolContext 负责提供依赖
ToolRegistry 负责统一管理和执行工具
AgentLoop 负责把工具能力交给模型，并处理模型发起的调用
```

## 二、推荐阅读顺序

### 1. 从简单工具开始：calculator.py

文件：

```text
app/tools/calculator.py
```

重点看：

- `CalculatorTool` 继承 `BuiltinTool`
- `name` 是工具唯一名称
- `description` 给模型判断何时调用
- `parameters` 描述模型需要传什么参数
- `read_only=True` 表示无副作用，可以并发
- `execute(params)` 是真正执行逻辑

先理解这个文件，因为它没有依赖注入、没有 UI、没有数据库，是最小闭环。

### 2. 再看抽象接口：base.py

文件：

```text
app/tools/base.py
```

重点看 `BuiltinTool` 的职责：

```text
BuiltinTool
    |
    +-- name
    +-- description
    +-- parameters
    +-- execute(params)
    +-- read_only
    +-- enabled
    +-- create(ctx)
    +-- to_openai_function()
    +-- cast_params()
    +-- validate_params()
```

这里要抓住两个点：

- `to_openai_function()` 把工具转换成 OpenAI function calling 格式。
- `cast_params()` 和 `validate_params()` 是执行前的统一参数处理。

### 3. 理解参数定义：schema.py

文件：

```text
app/tools/schema.py
```

它提供一个轻量 DSL，让工具参数不用手写复杂 JSON Schema。

示例：

```python
parameters = tool_params(
    "expression",
    expression=Str("要计算的数学表达式"),
)
```

生成的结构大致是：

```json
{
  "type": "object",
  "properties": {
    "expression": {
      "type": "string",
      "description": "要计算的数学表达式"
    }
  },
  "required": ["expression"]
}
```

学习重点：

- `Str / Int / Num / Bool / Arr / Obj` 分别对应 JSON Schema 类型
- `tool_params("a", "b", a=..., b=...)` 中位置参数表示必填字段
- 这个 schema 会被传给模型，也会被 `validate_params()` 使用
#### Tips
- DSL 是 Domain-Specific Language 的缩写，中文一般叫：
领域专用语言 意思是：专门为某个小领域设计的一套简化表达方式。
它不是一门新语言，只是用 Python 函数和类包装出一种更适合当前业务的写法。
dsl主要是方便开发人员，尤其是方便写内置工具的人。

### 4. 看依赖注入工具：notes.py

文件：

```text
app/tools/notes.py
```

这里可以看到工具如何拿到项目内部资源。

```text
ReadNotesTool
    |
    +-- create(ctx)
           |
           +-- ctx.note_mgr
```

重点看：

- 为什么 `ReadNotesTool` 需要 `note_mgr`
- 为什么覆盖 `create(ctx)`
- 创建笔记后为什么调用 `on_note_created`

这一步能理解：工具不是孤立函数，它可以通过 `ToolContext` 使用应用已有能力。
#### Tips
- cls()就是在实例化对象cls 代表“当前这个工具类”。
比如调用：
CalculatorTool.create(ctx)
那么：
cls == CalculatorTool
所以：
return cls()
等价于：
return CalculatorTool()

### 5. 看工具自动加载：loader.py

文件：

```text
app/tools/loader.py
```

内置工具加载：

```text
discover_builtin_tools()
    |
    +-- 扫描 app.tools 包
    +-- 跳过 base / schema / registry / loader 等基础模块
    +-- import 每个工具模块
    +-- 找 BuiltinTool 子类
```

注册流程：

```text
load_builtin_tools(ctx, registry)
    |
    +-- discover_builtin_tools()
    +-- tool_cls.create(ctx)
    +-- registry.register(tool)
```

这就是为什么新增内置工具时，通常只要在 `app/tools/` 下新建文件并继承 `BuiltinTool`。
#### Tips
- pkgutil 的定位很明确： 1 负责“找文件/找模块” 2 不负责“判断是不是工具” 
3不负责“注册工具” 4 不负责“执行工具”
### 6. 重点理解注册中心：registry.py

文件：

```text
app/tools/registry.py
```

这是工具系统的核心执行入口。

核心结构：

```text
ToolRegistry
    |
    +-- _tools: dict[str, BuiltinTool]
    |
    +-- register(tool)
    +-- unregister(name)
    +-- get(name)
    +-- get_all()
    +-- get_enabled()
    +-- get_openai_functions()
    +-- execute(name, params)
    +-- format_result(result)
```

执行流程：

```text
execute(name, params)
    |
    +-- 根据 name 找工具
    |      |
    |      +-- 找不到 -> TOOL_NOT_FOUND
    |
    +-- tool.cast_params(params)
    |
    +-- tool.validate_params(params)
    |      |
    |      +-- 校验失败 -> PARAM_INVALID
    |
    +-- _execute_with_retry(tool, params)
           |
           +-- tool.execute(params)
           +-- 异常分类
           +-- 可重试错误自动重试
```

错误分类：

```text
TOOL_NOT_FOUND  工具不存在
PARAM_INVALID   参数错误
TIMEOUT         超时
NETWORK         网络错误
PERMISSION      权限错误
RUNTIME         运行时错误
```

重试策略：

```text
TIMEOUT  -> 最多 3 次，间隔 1 秒
NETWORK  -> 最多 3 次，间隔 2 秒
RUNTIME  -> 最多 2 次，间隔 0.5 秒
```

这一步要重点理解：`ToolRegistry` 屏蔽了具体工具差异，让 AgentLoop 不关心工具是内置类还是用户脚本。

### 7. 看 Agent 如何使用工具：agent_loop.py

文件：

```text
app/core/agent_loop.py
```

关键流程：

```text
_state_stream()
    |
    +-- AIClient.chat_stream(ctx.messages, ctx.tools)
    +-- 收集模型返回的 tool_call

_state_execute()
    |
    +-- 生成 assistant tool_calls 消息
    +-- _partition_batches()
    +-- _execute_one() 或 _execute_concurrent_batch()
    +-- 追加 tool 消息

_state_feedback()
    |
    +-- 如果还没结束，继续下一轮模型调用
```

并发分区逻辑：

```text
[read_only, read_only, write, read_only]
        |
        v
[concurrent 前两个] -> [serial write] -> [concurrent 最后一个]
```

理解这里后，就知道模型为什么可以连续调用工具，以及工具结果如何进入下一轮推理。

### 8. 看用户脚本工具：script_adapter.py

文件：

```text
app/tools/script_adapter.py
```

用户脚本工具不是直接继承 `BuiltinTool`，而是被 `ScriptToolAdapter` 包装成统一接口。

```text
manifest.json
    |
    +-- name
    +-- description
    +-- parameters
    +-- dependencies
    +-- script
           |
           v
ScriptToolAdapter
    |
    +-- 像 BuiltinTool 一样注册
    +-- subprocess 执行 tool.py
    +-- stdin 传入 JSON 参数
    +-- stdout 最后一行解析为 JSON 结果
```

脚本协议：

```text
输入：
{"params": {...}, "context": {...}}

输出最后一行：
{"status": "success", "data": {...}}
```

重点理解：用户工具通过子进程隔离执行，避免脚本崩溃直接影响主程序。

### 9. 最后看 UI 管理：toolbox_panel.py

文件：

```text
app/ui/toolbox_panel.py
```

它主要负责用户脚本工具的管理：

```text
ToolboxPanel
    |
    +-- refresh()
    +-- _reload_user_tools()
    +-- _load_list()
    +-- _on_new_tool()
    +-- _on_generate_tool()
    +-- _on_edit()
    +-- _on_test()
    +-- _on_delete()
```

对应能力：

- 浏览工具
- 启用 / 禁用工具
- 新建工具
- AI 生成工具
- 编辑工具
- 测试工具
- 删除工具

测试工具时最终还是走：

```text
ToolTestDialog
    |
    +-- registry.execute(tool.name, params)
```

## 三、按问题驱动学习

学习时可以按这些问题逐个追踪代码：

### 问题 1：一个工具是如何被模型看到的？

追踪链路：

```text
BuiltinTool.parameters
    -> BuiltinTool.to_openai_function()
    -> ToolRegistry.get_openai_functions()
    -> AgentLoop 创建 TurnContext
    -> AIClient.chat_stream(messages, tools)
```

### 问题 2：模型传错参数怎么办？

追踪链路：

```text
ToolRegistry.execute()
    -> tool.cast_params()
    -> tool.validate_params()
    -> PARAM_INVALID
    -> ToolRegistry.format_result()
    -> tool 消息回传给模型
```

### 问题 3：为什么有些工具能并发，有些不能？

追踪链路：

```text
BuiltinTool.read_only
    -> AgentLoop._partition_batches()
    -> _execute_concurrent_batch()
    -> _execute_one()
```

判断原则：

```text
只读、无副作用：read_only=True
会写文件、改数据库、创建任务：read_only=False
```

### 问题 4：新增一个内置工具要改哪些地方？

通常只需要：

```text
1. 在 app/tools/ 下新建 xxx.py
2. 定义 class XxxTool(BuiltinTool)
3. 实现 name / description / parameters / execute
4. 如需依赖，覆盖 create(ctx)
```

不需要手动改注册表，因为 `loader.py` 会自动发现。

### 问题 5：用户脚本工具和内置工具有什么区别？

```text
内置工具：
    Python 类
    进程内执行
    可以直接使用 ToolContext 依赖

用户脚本工具：
    manifest.json + tool.py
    subprocess 子进程执行
    通过 stdin/stdout 通信
    由 ScriptToolAdapter 包装
```

注册后它们都进入 `ToolRegistry`，AgentLoop 不需要区分。

## 五、最小心智模型

最后可以把整个工具系统压缩成这个模型：

```text
工具定义层：
    BuiltinTool / ScriptToolAdapter

工具发现层：
    loader.py

工具管理层：
    ToolRegistry

模型调度层：
    AgentLoop

用户管理层：
    ToolboxPanel / ToolEditorDialog / ToolTestDialog
```

如果只记一个核心点：

```text
ToolRegistry 是中心。

模型看工具，通过它。
工具执行，通过它。
错误处理，通过它。
结果回传前格式化，也通过它。
```

当前项目中对于工具调用失败的处理是怎么做的？nanobot中又是怎么做的？

● 当前项目（assistant）的工具调用失败处理分三层：

  1. 工具级别 — 工具脚本自己返回错误结构体：
  {"status": "error", "data": {"message": "具体错误信息"}}
  包括超时、进程崩溃、依赖安装失败、JSON 解析失败等情况，都统一包装成这个格式。

  2. AgentLoop 级别 — 执行前校验 + 执行后格式化：
  - 参数校验失败 → 不执行工具，直接构造 error 结果返回给 LLM
  - 并发执行中异常 → catch 后包装为 error 结果
  - 通过 tool_event 信号通知 UI（"error" phase）

  3. 结果增强 — ToolManager.format_result() 在序列化时：
  - 检测到 status == "error" 自动追加提示：[工具执行出错。请分析错误原因，尝试不同的参数或方法。]
  - 这段提示作为 tool message content 的一部分发给 LLM，引导它自我修正

  关键点：工具失败不会中断 AgentLoop 状态机。错误结果作为 role: tool 消息追加到对话历史，LLM 在下一轮看到错误后自行决定重试或换方案。

  ---
  nanobot 的做法更激进一些：

  1. 相同点：错误也是作为 tool message 返回给 LLM，不中断循环
  2. 错误提示：同样追加 "[Analyze the error above and try a different approach.]"
  3. 额外机制：
    - 重复错误检测：AgentRunner 跟踪连续相同错误次数，超过阈值后强制停止循环，避免 LLM 无限重试同一个失败调用
    - SSRF/路径越界拦截：在 _run_tool() 中检测工具是否试图访问工作区外的路径或内网地址，直接拦截返回错误
    - 结果截断 + 持久化：超长结果写入临时文件，只返回摘要和文件路径给 LLM

  当前项目缺少的是 nanobot 的重复错误检测——如果 LLM 反复用相同参数调用同一个失败工具，当前实现会一直循环直到 max_turns
  耗尽。

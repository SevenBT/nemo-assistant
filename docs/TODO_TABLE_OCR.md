# 待办：表格识别（Table OCR）设计

> 状态：**待实现**（已暂缓，优先做「截图 → AI」主线）
> 记录时间：2026-06-12

## 背景

当前 OCR 用 `rapidocr_onnxruntime`，只能输出「带坐标的文本框」，**不支持结构化表格输出**。
已有的 `_reconstruct_layout`（`app/ui/screenshot_overlay.py`）做的是几何重建——
靠 x/y 对齐把文字排成行、按间隔插空格，对规整表格能输出「视觉对齐的纯文本」，
但不是真正的结构化表格（无单元格边界、不能导出 markdown/CSV、对合并单元格无能为力）。

## 核心决策

1. **独立按钮**，不并入「识字」。
   - 输出格式不同：文字 = 纯文本；表格 = 网格（markdown / CSV / HTML）。
   - 用户意图不同：让用户点「识表格」比程序自动判断更准、更可靠。
   - 契合「快捷动作」抽象：识字 / 识表格 / 问AI / 翻译 都是工具栏并列动作，共用框选→处理→输出管线。

2. **后端选型：AI 路径为主，离线几何兜底。**

## 实现方案对比（成本递增）

| 方案 | 依赖/体积 | 离线 | 精度 | 适用 |
|------|-----------|------|------|------|
| 几何升级版（扩展 `_reconstruct_layout`，按 x 聚类分列成网格 → markdown/TSV） | 零新依赖 | ✅ | 仅规整有线框表格；无边框/合并单元格/单元格内换行会失败 | 离线兜底 |
| **RapidTable**（`rapid_table`，slanet_plus 模型） | wheel 43kB + 模型 **~7MB**；依赖零增量（onnxruntime/opencv/numpy/Pillow 已被 RapidOCR 引入），仅多 requests/colorlog/omegaconf 几百 kB | ✅ | 中等偏上，输出带结构 HTML | 「既要离线又要高精度结构」时才引入 |
| **多模态大模型看图** | 复用「截图→AI」管线 | ❌ 需联网 | 最高，碾压乱/斜/合并/手写 | 主推 |

⚠️ **避坑：RapidTable 别选 `unitable` 模型**——要装 torch+torchvision + ~500MB 权重，
打包体积爆炸，违背桌面分发约束。要用就用默认 `slanet_plus`（ONNX，~7MB）。

## 推荐落地顺序

1. 「识表格」动作 → **优先走多模态 AI 路径**，输出 markdown 表（复用正在搭的截图→AI 管线，质量最高）。
2. 离线/无网络场景 → 补「几何升级版」作为降级方案。
3. RapidTable 作为中间选项，仅当「既要离线又要高精度结构」时再引入。

## 依赖前置

- 依赖「截图→AI」主线先把「快捷动作」抽象和 AI 调用管线搭好，表格只是其中一个预设动作。

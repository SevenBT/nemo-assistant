# 功能深化与扩展建议

> 生成日期：2026-06-28
> 基于对 `app/`(主程序)的只读代码走查。
> 定位锚点：**贴(便签) / 截(截图) / 调(对话调用工具)** 三条核心故事线，详见 `docs/整体计划.md`。

本文档只做评估与建议，不含任何代码改动。

---

## 一、三条核心线完成度速览

| 线 | 模块 | 完成度 | 主要扩展空间 |
|----|------|--------|-------------|
| 划词(贴) | `selection_controller` / `text_action_popup` / `result_bubble` | ~85% | 气泡内就地编辑、看完结果一键追问、识图与 OCR 连通、改写动作独立开关 |
| 截图(截) | `screenshot_controller` / `screenshot_overlay` / `vision_actions` | ~80% | 识图 prompt 自定义、OCR 结果→发给 AI、贴图标注 |
| 便签(贴/记) | `sticky_note_window` / `sticky_note_controller` / `notes_dialog` | ~70% | 颜色自选、便签标题编辑、划词存便签直接成 sticky、多便签恢复错位、新建后自动置前 |

整体看链路都已闭合，属于「能用但体验可再打磨」，不是半挂功能。

## 二、高优先级:能显著提升能力的深化点

### 1. 记忆 / 笔记的检索升级(RAG / 全文索引) ⭐⭐⭐⭐⭐
**现状**
- `memory_manager.build_memory_context` 注入记忆时只按 `importance DESC` 排序后截断到 4000 字符(`memory_manager.py:197`),**不按当前对话相关性检索**。记忆一多,高相关但低重要性的条目会被永久截掉。
- `note_manager.search_notes` 是朴素 SQL `LIKE '%kw%'`(`note_manager.py:611`),无全文索引、无相关性评分。
- 全库没有任何 embedding / 向量 / 语义检索代码。

**建议**
- 轻量方案:SQLite **FTS5** 做笔记/记忆全文检索,改动小、无新依赖。
- 进阶方案:`sqlite-vec` 或本地 embedding 做语义检索,把记忆注入从「全量截断」改为「按相关性 top-k」。
- 这同时打通了「整体计划」里待办的 **笔记问答(本地 RAG)**,是简历最大加分项。

### 2. NoteTool 能力补全 ⭐⭐⭐⭐
**现状**:`tools/notes.py` 只暴露 `list` 和 `create` 两个 action。AI 无法 update / delete / search / toggle 笔记;想搜笔记只能 list 全量再自己 filter。
**建议**:补 `search`(配合上面的 FTS5)、`update`、`delete`、`toggle_completed`。底层 `note_manager` 这些方法都已存在,只差工具层暴露。这直接强化「调」这条线对自有数据的操作能力。

### 3. 工具重试两条路径统一 ⭐⭐⭐⭐
**现状**(对应 `todos.md` 的已知问题):`registry._execute_with_retry`(`registry.py:203`)里,工具**抛异常**走 `_classify` 自动分类重试;工具**主动 `return {"status":"error"}`** 则依赖返回值里的 `error_type`/`retryable` 字段,而 `web_search`、`fetch_url` 等常常不设这些字段 → 两条路径重试行为不对称。
**建议**:对 return-error 路径也做一次错误分类(复用 `_classify` 的网络/超时判断),让两条路径的可重试判定一致。

### 4. 首次启动引导(Onboarding) ⭐⭐⭐⭐
**现状**:全库无 onboarding/wizard 代码。`main_window.py:259` 的首启判断只用于设窗口尺寸。新用户需自己摸索:进设置配 API Key、认识三个页面、发现热键。
**建议**:加一个首启向导:选后端 → 填 Key → 连接测试 → 三页功能 + 热键速览。影响「开箱即跑」第一印象,对作品集展示价值高。

## 三、中优先级:打磨现有体验

### 5. 划词结果气泡可交互
- `ResultBubble` 内容区是只读 `QTextEdit`(`result_bubble.py:260`),改写模式下用户不能就地微调 AI 结果,得切回主窗。
- oneshot(解释)看完结果想追问没有「在新会话继续」按钮。
- 建议:改写气泡允许就地编辑后再回填;结果气泡加一个「继续追问」入口。

### 6. 截图 OCR 与 AI 打通
- OCR 结果面板目前只「复制并关闭」(`screenshot_controller.py:68`),识字和识图是两条独立路径,看到 OCR 文字后无法直接追问 AI。
- 识图动作 prompt 硬编码在 `vision_actions.py:41`,不像划词那样支持用户自定义(划词已有 `selection_page` 自定义提示词)。
- 建议:OCR 面板加「发给 AI」;识图动作也接入自定义 prompt 设置。

### 7. 聊天气泡二级操作
- AI 气泡缺「复制全文 / 重新生成 / 另存便签」入口;代码块无语法高亮、无复制按钮(`_MessageText` 基于 `QTextBrowser.setHtml`)。
- `main_window.py:329` 的 `_tool_status` QLabel 是空占位,只被 `.hide()`,从未写入文字——可清理或利用。

### 8. Scheduler 任务管理补全
`scheduler.py` 缺 `update_job` / `enable_job` / `disable_job`(`jobs.json` 有 `enabled` 字段但无运行时接口),也无执行历史持久化。`list_scheduled_tasks` 工具未暴露 next_run_time。建议补这几个方法,让定时任务可改、可启停、可查历史。配合「整体计划」里的 **每日简报与主动性**。

### 9. Consolidator 摘要遗漏工具结果
`_messages_to_text`(`consolidator.py:75`)每条只取前 500 字符,且完全忽略 `tool` role 消息 → 长会话压缩后工具执行结果全丢,摘要质量受损。建议把 tool 结果纳入摘要输入。

### 10. 便签体验细节
- 颜色按全局索引循环(`sticky_note_window.py:63`),用户不能自选。
- 便签窗口无标题输入框,标题只能在笔记面板改。
- 划词「存便签」存成 `note_type="note"`(`selection_controller.py:237`),不是 `sticky`,无法直接贴桌面。
- `create_from_hotkey` 新建便签后无 `raise_()`+`activateWindow()`,可能不置前。
- 多个 pinned 便签恢复时用原坐标,无错位叠放,可能重叠。

## 四、低优先级 / 长期方向

### 11. nanobot 多渠道接入(差异化大方向) ⭐
`nanobot/` 是一个**完全独立**的生产级 agent 框架(独立 git 仓库、已发布 PyPI `nanobot-ai`),目前**未被主 app 引用**(`app/` 下无 `import nanobot`)。它的 `channels/` 有 19 个**真能用**的 IM 渠道实现(Telegram / Feishu / Slack / Discord / 微信 / 钉钉 / QQ / Signal …),并自带 OpenAI 兼容 API server。

- 价值:把桌面助手扩展为「多端可达」(手机 IM 也能调用同一助手),是与 Codex 形成品类差异的潜在大招。
- 成本:两套工具系统(同步 `BuiltinTool` vs 异步 `Tool`)、两套 registry 完全独立,集成需要架构决策。最小侵入路径是用 nanobot 的 API server 把主 app 封成一个 endpoint。
- 建议:作为独立大方向单独评估,不要轻易混入主 app。

### 12. LLM Gateway 健壮性
- 日志无轮转(`llm_gateway.py:626`),长跑会无限增长。
- 重试 `delays=(1.0, 2.0)` 无指数退避(`llm_gateway.py:517`)。
- `LocalRateLimiter` 仅全局 RPM,不支持 per-provider 差异化。
- 仅 LiteLLM 一个 adapter,所有请求都过这层中间层。

### 13. 其它零散项
- `agent_loop` `max_turns=10` 达上限时 `done` 信号 `error=None`(`agent_loop.py:357`),UI 看到 `ok=True` 却无输出,应给用户友好提示。`_MAX_WORKERS=4`、`max_turns` 等均硬编码,可配置化。
- 笔记 `recurrence` 字段已存库但无重复任务自动生成逻辑(`note_manager.py`);`todo` 类型笔记标为 Legacy 纯文本、无 checkbox 交互。
- 文件夹 `parent_id` 字段在但只支持单层,无递归查询。

## 五、推荐落地顺序

若要继续推进,建议按「先补能力短板,再打磨体验」:

1. **笔记/记忆 FTS5 检索 + NoteTool 补全**(#1 #2)——一次打通「第二大脑」,价值最高。
2. **工具重试路径统一**(#3)——修掉已知 bug,改动小。
3. **首次启动引导**(#4)——提升开箱体验,作品集友好。
4. **划词/截图体验打磨**(#5 #6 #7)——强化两条核心 demo 线。
5. **Scheduler 补全 + 每日简报**(#8)——做出助手的「主动性」。

> nanobot 多渠道(#11)单独立项评估,不在上述主线内。
